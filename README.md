# 金融規制ウォッチ（金融関連機関 新着モニター）

金融庁・日本銀行・日本取引所グループ(JPX)・日本証券業協会・JVCEA・日本公認会計士協会の公開情報を
**1日5回（JST 8:00 / 11:00 / 14:00 / 17:00 / 20:00）自動巡回**し、新着項目（日付・タイトル・リンク）を
一覧表示する静的サイト。新着があった時のみSlackへ通知。

## 仕組み
- `reg-monitor/crawler.js` … 全機関を巡回して `reg-monitor-site/data.json` を更新（既出URLと比較し新規のみ追記）。
- `reg-monitor-site/index.html` … `data.json` を読み込み、最新日付が上にくる一覧を表示。
- `.github/workflows/crawl.yml` … cron（UTC `0 23,2,5,8,11 * * *` = JST 8/11/14/17/20）で巡回 → data.jsonをコミット → GitHub Pagesへ公開。

## データ源
- 日本銀行 / JVCEA / 日本取引所グループ … 公式RSS（JPXは「マーケットニュース」`/rss/markets_news.xml`＝東証・OSE・TOCOMの上場/市場措置等、と「JPXニュース」`/rss/jpx-news.xml`の2フィード）
- 金融庁 / 日本証券業協会 / 日本公認会計士協会 … 公開ページのHTML解析
- ※サイトには複数のニュース系統があり得る（例: JPXはコーポレートのプレスリリースとは別に「マーケットニュース」がある）。当局を追加・見直す際は、トップページのRSSアイコンや新着欄を確認し、主要系統を取りこぼさないこと。RSSがあればHTML解析より堅牢。

## 巡回頻度のルール（恒久仕様・維持すること）
- **全組織を1日複数回（現状5回）巡回**する。巡回は機関単位ではなくワークフロー単位で、1回の実行で `crawler.js` の `SITES` 全件をループ取得する。
- したがって **`SITES` に追加する将来の組織も、自動的に同じ頻度で監視対象**になる。組織追加は `SITES` 配列に1行足すだけで、頻度設定（crawl.yml の cron）は変更不要。
- 朝1回だけだと、朝の巡回後（当日昼〜午後）に公表される新着（金融庁の処分・パブコメ、日銀の金融政策決定会合 等）を翌朝まで取りこぼすため、複数回巡回は必須。**頻度を朝1回などに戻さないこと。**
- 注意: 「新規0件」を見て"新着が無い"と即断しない。各当局の実新着と突き合わせて検証する（取得タイミングのズレで一時的に0になることがある）。

## 新しい組織の追加手順
1. `reg-monitor/crawler.js` の `SITES` に1行追加。type の選び方（堅牢な順）:
   - **`rss-index`（最優先）**… その機関が「RSS一覧（ハブ）ページ」を持つ場合は、ハブのURLを指定する。**毎回ハブを見て子RSSを全て自動発見し巡回**するので、機関がフィードを増減しても自動追従する（例: JPXの `rss/index.html` → マーケットニュース/売買停止/注意喚起 等を全取得）。RSSページがある組織は必ずこの方式にする。
   - `rss` … 単一のRSSフィードしかない場合。
   - `html` / `fsa` … RSSが無くHTML解析する場合（レイアウト変更で要保守）。
   - 任意: `excludeFeeds:['site-updates']` のように、`rss-index` でノイズの多い子フィードを除外可能。
2. 必要なら `reg-monitor-site/index.html` の `AGENCY_ORDER`（表示順）に機関名を追加。
3. 以上だけで、1日5回の巡回・新規差分検知（同一URLの日付更新も再検知）・Slack通知・一覧表示がすべて自動適用される。

### 【恒久ルール】RSSハブの自動巡回
機関が「RSS一覧ページ」を持つ場合は、個別フィードをハードコードせず **`rss-index` でハブを毎日巡回し、子RSSを全て自動で辿る**こと。これは現行・将来追加のすべての組織に適用する（RSSページがあるのに一部フィードしか見ない、という取りこぼしを防ぐ）。

### 【恒久ルール】ノイズ除外
実務的に価値の無い文字列（「〜のページを更新しました」等のサイト保守・データページ定期更新の通知）は、**現行・将来追加の全機関で取り込まない**。2系統で除外:
- フィード単位: `rss-index` の `excludeFeeds:['site-updates']` 等で、保守系の子フィード自体を巡回対象から外す（例: JPX「サイト更新情報」）。
- タイトル単位: `crawler.js` の `NOISE_TITLE`（`isNoise()`）で無価値パターンを除外。既に蓄積済みのノイズも毎回の巡回で除去される。新たなノイズ文言が出たら `NOISE_TITLE` に追記する（取りこぼし防止のため保守的に＝明確な無価値パターンのみ。定例レポートの実更新等は残す）。

## 法令ビューアとの双方向連携（lawrefs）
各ニュースに「どの法令の・どの条の・どんな動向か」を意味づけして、[法令ビューア](https://finoject.github.io/finoject-law-viewer/)と双方向に連携する。
- **共有データ契約**: `crawler.js` が `reg-monitor/enrich.js` の `buildLawrefs()` で各itemに `lawrefs = [{ id, art, kind, label }]` を付与し `data.json` に保存。`id`=law_id、`art`=「第N条」（算用数字。ビューアの表記に一致）、`kind`=パブコメ/改正/公布/ガイドライン/告示/通達等/施行/その他。
- **本文/PDF解析**: タイトルだけでは法令・条が判らないことが多いため、`worthFetching` なニュースはリンク先HTML＋PDF（`pdftotext`。`crawl.yml` に `poppler-utils`）まで読んで法令名・条番号を抽出する。1巡回あたりの取得上限は `MAX_FETCH_PER_RUN`（既定40件、未処理は次回以降）。一度処理した項目は `enriched` を立て再取得しない（更新時は再算出）。
- **法令名マッチ**: `LAW_DICT` を長い名称優先でマッチ（「○○法施行令」を「○○法」と誤認しない）。条番号は前方80字以内の最も近い法令名に紐付け（離れた条は安全側で捨てる）。新しい法令を足すときは `LAW_DICT` に名称を追加するだけで、📖チップとビューア側パネル/バッジの両方が自動で効く。
- **表示**: reg-monitor-site は `lawrefs` から「📖 法令名（N条）」チップ（法令単位に集約・種別バッジ色分け）。法令ビューアは `lawrefs` を逆引きして「📡 この法令の最近の規制動向」パネルと条文バッジを表示。
- 限界: 複数法令が混在する新旧対照表PDFでは条の帰属がまれに不正確になりうる（その場合は紐付け窓80字を狭める）。

## 手動実行
GitHubの「Actions」タブ →「crawl-and-publish」→「Run workflow」で即時実行可能。
ローカルでは `cd reg-monitor && npm install && node crawler.js`。
